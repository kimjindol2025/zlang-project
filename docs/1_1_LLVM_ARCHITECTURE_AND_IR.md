# 🏛️ LLVM 전설: 1.1 컴파일러 아키텍처와 IR(Intermediate Representation)

## 📚 강의 개요

**LLVM(Low Level Virtual Machine)** 은 단순히 컴파일러가 아니라, **'재사용 가능한 컴파일러 구성 요소의 집합'** 입니다.

우리가 만든 Z-Lang이 수많은 CPU(x86, ARM, RISC-V)에서 돌아가게 만드는 **마법의 도구** 입니다.

---

## 1️⃣ 컴파일러의 3단계 구조

현대적인 컴파일러는 역할을 셋으로 나눕니다. **LLVM은 이 중 중간과 끝을 완벽하게 책임집니다.**

```
【 전통적인 컴파일러 아키텍처 】

┌─────────────────────────────────────────────────────────┐
│                     SOURCE CODE                          │
│              (Zig, C, Z-Lang, ...)                       │
└────────────────────┬────────────────────────────────────┘
                     ↓
         ┌───────────────────────┐
         │    FRONT-END          │
         ├───────────────────────┤
         │ • 렉서 (Lexer)         │
         │ • 파서 (Parser)        │
         │ • 의미분석 (Semantics) │
         └────────────┬──────────┘
                      ↓
          ┌───────────────────────┐
          │   LLVM IR 생성        │
          │ (In-memory, .ll, .bc) │
          └────────────┬──────────┘
                       ↓
        ┌──────────────────────────┐
        │    OPTIMIZER             │
        ├──────────────────────────┤
        │ • 상수 폴딩 (Const Fold)  │
        │ • 데드 코드 제거         │
        │ • 인라인 최적화          │
        │ • 루프 언롤링            │
        └────────────┬─────────────┘
                     ↓
        ┌──────────────────────────┐
        │    BACK-END              │
        ├──────────────────────────┤
        │ • 레지스터 할당          │
        │ • 기계어 선택            │
        │ • 스케줄링               │
        └────────────┬─────────────┘
                     ↓
         ┌───────────────────────┐
         │  MACHINE CODE         │
         │  (x86, ARM, RISC-V)   │
         └───────────────────────┘
```

### 📌 세 가지 역할

#### **Front-end (프론트엔드)**
```
역할: 소스 코드를 읽어 의미를 분석
출력: LLVM IR

구현: 각 언어별로 다름
  ├─ Clang: C/C++의 프론트엔드
  ├─ rustc: Rust의 프론트엔드
  ├─ Swift Compiler: Swift의 프론트엔드
  └─ 우리: Z-Lang의 프론트엔드 (만들어야 함!)
```

#### **Optimizer (중간)**
```
역할: IR을 분석하여 최적화
입력: LLVM IR
출력: 최적화된 LLVM IR

특징: 언어에 무관함!
  ├─ C, C++, Swift, Rust 모두 같은 최적화기 사용
  └─ 새로운 최적화가 모든 언어에 즉시 적용됨
```

#### **Back-end (백엔드)**
```
역할: 최적화된 IR을 기계어로 변환
입력: LLVM IR
출력: 네이티브 코드 (x86-64, ARM64, RISC-V, ...)

특징: 아키텍처 의존적
  ├─ 각 CPU마다 구현 필요
  ├─ LLVM은 15+ 주요 아키텍처 지원
  └─ 새 언어는 프론트엔드만 만들면 모든 CPU 지원!
```

## 2️⃣ LLVM IR: 전설의 공용어

### 🎯 IR이란?

LLVM IR은 **하이레벨 언어보다는 낮고, 기계어보다는 높은 '추상화된 어셈블리'** 입니다.

```
Zig, C, Rust, Python, ...
          ↓
    ────────────  (언어 독립적 경계선)
          ↓
      LLVM IR  ← 모든 언어의 공용어
          ↓
    ────────────  (타겟 독립적 경계선)
          ↓
x86-64, ARM64, RISC-V, WebAssembly, ...
```

### 🔧 IR의 특징

```
【 LLVM IR의 핵심 특징 】

1. 무한한 가상 레지스터
   └─ 실제 CPU 레지스터 개수에 제한 없음
   └─ 나중에 백엔드가 실제 레지스터로 할당

2. 엄격한 타입 시스템
   └─ i32, i64, f64, ptr, ... 모든 값에 명시적 타입
   └─ 최적화와 안전성을 위함

3. SSA (Static Single Assignment)
   └─ 모든 변수는 단 한 번만 할당됨
   └─ 최적화 알고리즘이 수학적으로 명확함

4. Control Flow Graph (CFG)
   └─ 함수를 기본 블록(BasicBlock)들의 그래프로 표현
   └─ 분기와 루프를 명시적으로 표현
```

### 📋 IR의 세 가지 형태

```
【 LLVM IR 표현 방식 】

1. In-memory (메모리 형식)
   ├─ LLVM C++ 클래스로 표현
   ├─ 프로그래밍으로 생성/분석
   └─ 가장 빠름 (컴파일러가 주로 사용)

2. Bitcode (.bc 파일)
   ├─ 효율적인 바이너리 형식
   ├─ 저장/전송에 최적화
   └─ 크기: 텍스트의 1/10

3. Assembly (.ll 파일)
   ├─ 인간이 읽을 수 있는 텍스트 형식
   ├─ 우리가 디버깅/학습할 때 봄
   └─ 쉽게 읽고 쓸 수 있음
```

### 🔄 상호 변환

```
C 소스 코드
    ↓ (Clang 파서)
In-memory IR
    ↓ (IR → Bitcode)
.bc 파일
    ↓ (또는 IR → Assembly)
.ll 파일 (우리가 읽을 수 있음!)
    ↓ (백엔드)
Native Code (.o, .exe, ...)
```

## 3️⃣ LLVM IR 맛보기 (함수 정의)

### 📝 간단한 예제

#### **Zig 코드**

```zig
fn add(a: i32, b: i32) -> i32 {
    return a + b;
}
```

#### **LLVM IR 표현**

```llvm
; 두 정수를 더하는 함수 (LLVM IR 표현)
define i32 @add(i32 %0, i32 %1) {
entry:
    %2 = add i32 %0, %1
    ret i32 %2
}
```

### 🔍 각 부분 설명

```
define i32 @add(i32 %0, i32 %1) {
│      │   │              │
│      │   │              └─ 두 번째 인수 (32비트 정수)
│      │   │
│      │   └─ 첫 번째 인수 (32비트 정수)
│      │
│      └─ 함수 이름 (@add)
│
└─ 반환 타입 (i32 = 32비트 정수)

entry:
  └─ 기본 블록의 이름 (첫 번째 블록)

%2 = add i32 %0, %1
│    │   │    │   └─ 오른쪽 피연산자 (%1)
│    │   │    │
│    │   │    └─ 왼쪽 피연산자 (%0)
│    │   │
│    │   └─ 연산: add (덧셈)
│    │
│    └─ 피연산자들의 타입 (i32)
│
└─ 결과를 저장하는 가상 레지스터 (%2)

ret i32 %2
    │   └─ 반환할 값 (%2)
    │
    └─ i32로 반환
}
```

### 📊 더 복잡한 예제

#### **루프가 있는 함수**

```zig
fn sum_to_n(n: i32) -> i32 {
    let mut sum: i32 = 0;
    let mut i: i32 = 0;
    while i < n {
        sum = sum + i;
        i = i + 1;
    }
    return sum;
}
```

#### **LLVM IR (간단히)**

```llvm
define i32 @sum_to_n(i32 %n) {
entry:
  br label %loop                    ; 루프로 분기

loop:
  %i = phi i32 [0, %entry], [%i_next, %loop]      ; phi 노드!
  %sum = phi i32 [0, %entry], [%sum_next, %loop]

  %cond = icmp slt i32 %i, %n      ; i < n 비교
  br i1 %cond, label %body, label %exit

body:
  %sum_next = add i32 %sum, %i     ; sum = sum + i
  %i_next = add i32 %i, 1          ; i = i + 1
  br label %loop                    ; 루프로 돌아가기

exit:
  ret i32 %sum                      ; 반환
}
```

## 🔬 전공 심화: SSA (Static Single Assignment)

### 📌 SSA란?

```
【 SSA의 정의 】

"프로그램의 모든 변수는 정확히 한 번만 할당된다"

이는 수학적 이점을 제공합니다:
  ├─ 데이터 흐름 분석이 명확함
  ├─ 최적화 알고리즘이 단순함
  └─ 검증이 용이함
```

### 🎯 SSA가 필요한 이유

```c
// 전통적인 코드 (Non-SSA)
int x = 5;      // x의 첫 번째 할당
x = x + 1;      // x의 두 번째 할당
x = x * 2;      // x의 세 번째 할당
return x;       // x의 최종값은?

// 최적화기의 관점:
// x의 최종값이 무엇인지 추적하기 어려움!
// 중간 계산을 버릴 수 있는지 알 수 없음!
```

### ✨ SSA 변환

```llvm
; SSA 형식으로 변환
%x1 = 5             ; 첫 번째 값
%x2 = add i32 %x1, 1 ; 두 번째 값
%x3 = mul i32 %x2, 2 ; 세 번째 값
ret i32 %x3

; 최적화기의 관점:
; %x1은 단 한 번만 할당됨
; %x2는 %x1과 1로부터 정의됨
; %x3은 %x2와 2로부터 정의됨
; 추적이 명확!
```

### 🔄 루프와 Phi 노드

SSA에서 루프는 어떻게 표현할까?

```zig
// 루프 (변수가 여러 번 할당됨)
let mut x = 0;
while condition {
    x = x + 1;
}
return x;
```

SSA에서는 **phi 노드** 를 사용합니다:

```llvm
loop:
  %x = phi i32 [0, %entry], [%x_next, %loop]
  ; %x는:
  ;   - %entry 블록에서 오면 0
  ;   - %loop 블록에서 오면 %x_next

  %cond = icmp ...
  br i1 %cond, label %body, label %exit

body:
  %x_next = add i32 %x, 1
  br label %loop  ; 루프로 돌아가기
```

**Phi 노드의 의미**: "어떤 블록에서 왔는지에 따라 다른 값을 선택"

## 📦 Zig와 LLVM의 연결고리

### 🔗 Zig의 컴파일 과정

```
당신의 Zig 코드
    ↓ (Zig 프론트엔드)
  Zig AST (추상 구문 트리)
    ↓ (Zig 의미분석)
  LLVM IR 생성 (LLVM C API 사용)
    ↓ (LLVM 최적화)
  최적화된 IR
    ↓ (LLVM 백엔드)
  네이티브 코드 (.o)
    ↓ (링커)
  최종 바이너리
```

### 🎯 Zig가 LLVM을 사용하는 이유

```
【 Zig가 LLVM을 선택한 이유 】

1. 성능
   └─ 직접 기계어를 만드는 것보다 LLVM의 최적화가 더 좋음

2. 재사용성
   └─ 기존의 검증된 최적화 기술들을 모두 활용

3. 다중 플랫폼
   └─ x86, ARM, RISC-V, WebAssembly, ... 모두 지원

4. 품질
   └─ LLVM 프로젝트의 끊임없는 개선로부터 이득

5. 생산성
   └─ 백엔드 구현에 시간을 낭비하지 않고 언어 설계에 집중
```

```
   c) 검증 가능성
      - IR이 기계어보다 검증하기 쉬운가?
      - SSA 형식이 어떻게 도움이 되는가?

3. 2-3 페이지의 분석 보고서 작성

4. 예상 답변 요소:
   - IR을 거치면 각 언어는 프론트엔드만 구현하면 됨
   - LLVM의 통합 최적화가 모든 언어에 동시에 적용됨
   - 새 CPU 추가는 백엔드만 구현하면 됨 (프론트엔드 재작성 불필요)
   - 최적화 알고리즘이 SSA 형식에서 수학적으로 명확함
```
